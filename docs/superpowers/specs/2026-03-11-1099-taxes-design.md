# 1099-NEC Tax Tracking — Design Spec

**Date:** 2026-03-11
**Status:** Approved
**Project:** Got Key'd Realty Commission Tracker

## Overview

Add a "Taxes" section to the commission tracker that automatically accumulates agent commission data by calendar year and generates filled IRS 1099-NEC PDFs. When a commission invoice is generated, the agent's gross commission (agent_share) is auto-added to their tax record for that calendar year. Manual adjustments are supported for corrections or outside-system payments.

## Business Rules

- **1099-NEC** is the correct form for real estate agent nonemployee compensation
- Report the **full `agent_share`** field from the transaction (gross commission before compliance fee deductions, NOT `total_payout`) per Michigan law — agents deduct compliance fees as business expenses
- Calendar year aggregation (Jan 1 – Dec 31), NOT anniversary-based cap years
- $600 reporting threshold (2025+); configurable if legislation changes
- January 31 filing deadline for both federal and Michigan state
- Michigan requires direct 1099-NEC filing with Treasury (not through federal CF/SF program)
- Missing agent tax info (TIN, address) does NOT block generation — blank fields are left empty on the PDF

## Database Changes

### Migration strategy

The `_create_tables()` method already uses `CREATE TABLE IF NOT EXISTS`. For new columns on the existing `agents` table, add an `_migrate_schema()` method called after `_create_tables()`. It checks for column existence using `PRAGMA table_info(agents)` and runs `ALTER TABLE agents ADD COLUMN ...` for any missing columns. This is safe and idempotent.

### Agent table additions (all nullable)

| Column | Type | Purpose |
|--------|------|---------|
| tin | TEXT | SSN or EIN |
| street_address | TEXT | Mailing address line |
| city | TEXT | City |
| state | TEXT | State (default "MI") |
| zip_code | TEXT | ZIP code |

**Security note:** TINs (SSNs/EINs) are stored as plaintext in SQLite. This is an accepted risk for this single-user desktop application where the database file lives in the user's app data directory. If multi-user access is added in the future, TIN encryption should be revisited.

### New `tax_records` table

| Column | Type | Purpose |
|--------|------|---------|
| id | INTEGER PK | Auto-increment |
| agent_id | INTEGER FK | Links to agents |
| tax_year | INTEGER | Calendar year |
| total_compensation | REAL | Sum of agent_share from transactions |
| manual_adjustment | REAL | Override delta (default 0.0) |
| adjustment_note | TEXT | Reason for manual adjustment |
| filed | BOOLEAN | Whether 1099 was generated |
| filed_date | TEXT | When it was filed |

**Unique constraint:** (agent_id, tax_year)

**Effective 1099 amount** = total_compensation + manual_adjustment

### app_settings addition

- Key: `company_ein`, Value: company's EIN string

### Methods requiring updates in database.py

- `insert_agent()` — add new columns to INSERT statement
- `_row_to_agent()` — map new columns to Agent dataclass fields
- `update_agent()` — add new columns to UPDATE statement
- New methods: `get_tax_record()`, `upsert_tax_record()`, `get_tax_records_for_year()`, `update_tax_record_adjustment()`, `mark_tax_record_filed()`

### Methods requiring updates in agent_manage_frame.py

- The raw SQL `UPDATE agents SET ...` query must include the new columns

## Model Changes

### Agent dataclass additions

- `tin: str = ""`
- `street_address: str = ""`
- `city: str = ""`
- `state: str = "MI"`
- `zip_code: str = ""`

- `id`, `agent_id`, `agent_name`, `tax_year`, `total_compensation`, `manual_adjustment`, `adjustment_note`, `filed`, `filed_date`
- Property: `effective_amount` = total_compensation + manual_adjustment

## UI Changes

### Agent Profile — Tabbed Interface

The agent manage frame gets a two-tab layout:

**Tab 1 — "Profile"** (all existing fields):
- Basic Info (name, license number, license expiration)
- Commission Structure (split type, percentages, tiers, transaction fee)
- Cap Settings (cap amount, contract date, cap paid adjustments)
- Notes

**Tab 2 — "Tax Info"** (new fields):
- TIN section: SSN or EIN entry with SSN/EIN toggle
- TIN displayed masked (***-**-1234) after entry, "Show" toggle to reveal
- Mailing Address: street, city, state, zip
- Filing status indicators per tax year

All fields optional.

### Sidebar Navigation

New "Taxes" item in sidebar, below "History" and above any future sections.

### Taxes Dashboard (TaxesFrame)

**Top section:**
- Year selector dropdown (defaults to current year)
- Company EIN display with edit button (first-visit setup prompt)
- Summary stats: total agents with earnings, total compensation paid, agents above threshold, agents with missing tax info

**Agent table:**
- Columns: Agent, Total Compensation, Adjustment, 1099 Amount, Status, Actions
- Status badges (priority order, highest first): "Filed" (blue), "Missing TIN"/"Missing Address" (yellow), "Below Threshold" (gray), "Ready" (green). Only one badge shown per agent — the highest priority match.
- Row actions: "Adjust" button, "Generate 1099" button
- Bulk action: "Generate All 1099s" button for all eligible agents

**Adjustment modal:**
- Shows current auto-tracked total
- Adjustment amount field (positive or negative)
- Note field (required)

## Auto-Tracking Integration

### On invoice generation (app.py)

After a transaction is committed to the database:
1. Extract calendar year from `closing_date`
2. Look up or create `tax_records` row for (agent_id, tax_year)
3. Add transaction's `agent_share` to `total_compensation`

**Note:** Transaction deletion does not currently exist in the codebase. If added in the future, the delete logic should subtract the corresponding `agent_share` from the tax record. This is out of scope for this feature.

## 1099-NEC PDF Generation

### Template

Uses the uploaded `f1099nec.pdf` fillable IRS form. The template must be moved to `commission_tracker/assets/f1099nec.pdf` and added to the PyInstaller spec's `datas` list so it is bundled in the .app. At runtime, accessed via `get_resource_path('assets/f1099nec.pdf')`.

Filled via PyMuPDF (fitz) form field API.

### Field mapping

| PDF Field | Data Source |
|-----------|------------|
| f1_1 | Tax year (e.g., "2025") |
| f1_2 | Company name and address from `constants.py` (COMPANY_NAME, COMPANY_ADDRESS) |
| f1_3 | Company EIN (from app_settings) |
| f1_4 | Agent TIN (blank if missing) |
| f1_5 | Agent name |
| f1_6 | Agent street address (blank if missing) |
| f1_7 | Agent city, state, zip (blank if missing) |
| f1_9 | Box 1: Nonemployee compensation amount |

State tax fields (boxes 5-7) left empty unless state withholding is added later.

### Copies generated

- **Copy B** (recipient — give to agent)
- **Copy 2** (payer records — keep on file)
- Copy A and Copy 1 not generated (filed electronically or on red-ink forms)

### Re-generation workflow

A 1099 can be re-generated at any time (even after being marked as filed). Re-generating updates the `filed_date` and overwrites the previous PDF. The `filed` flag is set to true on generation. If an adjustment is made after filing, the status reverts to "Ready" so the user knows to re-generate.

### Output path

1099 PDFs are saved to a `1099/{year}/` subdirectory inside the platform app data directory (same parent as the database): `{get_data_dir()}/1099/{year}/{agent_name}_1099NEC_{year}.pdf`. The user is shown the output path after generation and can open the containing folder.

### Missing data handling

If TIN, address, or EIN is missing, the PDF is still generated with those fields left blank. A warning is shown but generation is not blocked.
